# Предиктивная аналитика группы риска (Early Warning / Retention)

Система раннего выявления студентов с **риском отчисления** для университета.

> ⚠️ **Это не приговор, а сигнал.** Модель только **сигнализирует** о вероятном
> риске и объясняет, почему. Решение и любое вмешательство — **всегда за
> человеком** (эдвайзером/деканатом). Список риска чувствителен: доступ
> ограничен ролью и логируется, персональные данные не выходят за пределы сети.

## Что это и что это НЕ

- **Ядро — классическая ML-модель** (gradient boosting, LightGBM) на табличных
  данных. **GPU не нужна.**
- **LLM (Qwen3-14B через OVMS) используется ТОЛЬКО** для оформления текста
  уведомлений эдвайзерам. На прогноз риска она **не влияет**; при недоступности
  LLM работает локальная заглушка.
- Весь пайплайн работает **end-to-end на синтетике**, без реального Платонуса и
  без живого LLM — чтобы всё проверить до подключения боевых источников.

## Архитектура

```
Платонус (MySQL, read-only)               ┌──────────────┐
        │  SQL из data_queries.yaml       │   n8n        │ еженедельно
        ▼                                 │  (cron)      │
  platonus.py ──► features.py ──► model.py│      │       │
 (или синтетика)  (срез без утечки)  LightGBM    ▼       │
        │                            │     POST /score/run
        ▼                            ▼     POST /notify/run
   data/raw/*.csv              Postgres / SQLite      │
                              (признаки, прогнозы,     ▼
                               журнал доступа)    эдвайзеры (email)
        ▲                            │
        │                            ▼
        └────────  FastAPI (app.py)  ◄──── Next.js дашборд (серверный прокси)
```

- **Слой данных** (`platonus.py`, `data_queries.yaml`, `synthetic_data.py`).
- **Признаки** (`features.py`) — временной срез на неделю отсечки **без утечки**.
- **Модель** (`model.py`, `train.py`) — LightGBM + калибровка + вклад факторов.
- **Скоринг** (`scoring.py`) — батч-прогон, запись прогнозов в БД (`storage.py`).
- **Бэкенд** (`app.py`) — FastAPI для дашборда и n8n.
- **Фронтенд** (`frontend/`) — Next.js 15 (App Router) дашборд эдвайзеров.

## Стек

- Backend: Python 3.11+, FastAPI, LightGBM, scikit-learn, pandas.
- Хранилище: Postgres (прод) или SQLite (по умолчанию для разработки/демо).
- Источник данных: Платонус (MySQL, read-only) или синтетика (CSV).
- LLM: Qwen3-14B через OVMS (OpenAI-совместимый API) — только текст уведомлений.
- Оркестрация: n8n (еженедельный прогон + рассылка).
- Frontend: Next.js 15 + TypeScript + Tailwind + UI-примитивы в духе shadcn/ui.

Все адреса/токены — **только через переменные окружения** (см. `.env.example`).

## Быстрый старт (на синтетике, без Платонуса и без LLM)

### 1. Бэкенд: окружение и зависимости

```bash
cd backend
python -m venv .venv
# Windows:  .venv\Scripts\activate     |  Linux/Mac:  source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env        # при желании отредактируйте; по умолчанию всё работает
```

По умолчанию `DATA_SOURCE=synthetic` и `DATABASE_URL` пуст → используется
локальный **SQLite** (`backend/data/risk.db`). Postgres поднимать не обязательно.

### 2. Весь цикл одной командой

```bash
python run_pipeline.py
```

Это: генерирует синтетику → строит признаки → обучает модель → выводит метрики →
прогоняет батч-скоринг → печатает пример списка риска.

### 3. Или по шагам

```bash
python synthetic_data.py      # 1. синтетический датасет в data/raw/*.csv
python train.py               # 2. обучение модели + метрики (артефакт в data/model/)
python evaluate.py            # 3. отдельный отчёт по качеству модели
python scoring.py             # 4. батч-скоринг → прогнозы в БД
```

### 4. Запуск API

```bash
uvicorn app:app --host 0.0.0.0 --port 8000
```

Проверка эндпоинтов:

```bash
curl http://localhost:8000/health
curl "http://localhost:8000/risk?faculty=Факультет%20ИТ"
curl http://localhost:8000/student/S100411
curl http://localhost:8000/summary
# защищённые операции (если задан SERVICE_TOKEN — добавьте заголовок):
curl -X POST http://localhost:8000/score/run -H "X-Service-Token: <TOKEN>" -d '{}'
curl -X POST http://localhost:8000/train     -H "X-Service-Token: <TOKEN>" -d '{}'
curl -X POST http://localhost:8000/notify/run -H "X-Service-Token: <TOKEN>" -d '{"limit":50}'
```

### 5. Фронтенд

```bash
cd frontend
cp .env.example .env.local    # укажите RISK_API_BASE (адрес бэкенда)
npm install
npm run dev                   # http://localhost:3000
```

Дашборд: список группы риска с фильтрами, карточка студента с факторами, сводка
по факультетам/группам. Браузер ходит **только** в серверный прокси Next
(`/api/*`), который уже обращается к бэкенду — адрес бэкенда и токен наружу не
отдаются.

## Подключение реального Платонуса

1. Откройте `backend/data_queries.yaml` и **впишите реальные имена таблиц/полей**
   своей инсталляции Платонуса вместо плейсхолдеров `<...>`. Каждый запрос должен
   вернуть колонки с указанными псевдонимами (`AS ...`) — код опирается на них,
   а не на имена ваших таблиц. Запросы строго **read-only (SELECT)**.
2. В `backend/.env` задайте:
   ```
   DATA_SOURCE=platonus
   PLATONUS_HOST=...  PLATONUS_PORT=3306  PLATONUS_DB=...  PLATONUS_USER=...  PLATONUS_PASSWORD=...
   PLATONUS_TERMS=2024-1,2024-2,2025-1     # какие учебные периоды выгружать
   ```
3. Дальше всё то же: `python train.py`, `python scoring.py`. Слой признаков и
   модель от источника данных не зависят.

> Без персональных данных наружу: тяните минимум (стабильный внутренний
> `student_id`, без ИИН/телефонов/адресов).

## Подключение Postgres (прод)

```bash
docker compose up -d postgres          # поднимет Postgres и применит миграции
# в backend/.env:
DATABASE_URL=postgresql://risk:risk@localhost:5432/risk_group
```

Схема таблиц — в [`backend/migrations/001_init.sql`](backend/migrations/001_init.sql)
(`features`, `predictions`, `score_runs`, `access_log`). При SQLite те же таблицы
создаются автоматически (`storage.py`).

## Связка с n8n (еженедельный прогон + алерты)

Импортируйте [`n8n/risk_group_weekly.json`](n8n/risk_group_weekly.json). Воркфлоу:

1. **Cron** — каждый понедельник 06:00.
2. **POST `/score/run`** — батч-скоринг текущих студентов.
3. **POST `/notify/run`** — по флагам генерирует короткий текст «на что обратить
   внимание» (Qwen3 или заглушка) и возвращает список для рассылки.
4. **Split + Email** — рассылка эдвайзерам.

Переменные окружения n8n: `RISK_API_BASE`, `RISK_SERVICE_TOKEN` (= `SERVICE_TOKEN`
бэкенда), `ADVISOR_EMAIL`. Вебхук-эндпоинты для интеграции — это `POST /score/run`
и `POST /notify/run` (оба защищены заголовком `X-Service-Token`).

## Признаки и отсутствие утечки данных (важно)

Признаки строятся на **состояние студента до недели отсечки** `CUTOFF_WEEK`
(по умолчанию 6-я неделя семестра). В признаки попадают **только** события с
`week <= CUTOFF_WEEK`. Целевая метка — **отчисление строго позже** отсечки
(`week > CUTOFF_WEEK`). Студенты, выбывшие до отсечки, из скоринга исключаются.
Так в признаках нет будущих данных. Каждый признак и его смысл задокументированы
в `FEATURE_DOC` (`features.py`).

Примеры признаков: средний балл и его динамика, минимальный балл (провал),
доля пропусков и её тренд, число несданных/просроченных, пересдачи, провалы по
ключевым предметам, активность в LMS и её тренд, учебная нагрузка, демография.

## Модель и метрики

- **LightGBM**, дисбаланс классов — через `scale_pos_weight = нег/поз`.
- **Валидация по когортам во времени**: обучение на ранних учебных периодах,
  проверка на поздних (не случайный train/test split).
- **Калибровка вероятностей** (isotonic) — чтобы «риск = 0.7» читался как
  вероятность.
- **Объяснимость**: вклад факторов через LightGBM `pred_contrib` (точные значения
  Шепли для деревьев) — эдвайзер видит, *почему* студент во флаге.
- **Настраиваемый порог**: флаг получают верхние `RISK_TOP_FRACTION` (топ-X%) по
  вероятности — управление нагрузкой эдвайзеров.

Метрики (`evaluate.py`) — **не accuracy** (классы несбалансированы), а: **PR-AUC**
(average precision), ROC-AUC, **recall/precision по группе риска при пороге**,
**recall@k** (топ-5/10/20%), **калибровка** по бинам.

Пример отчёта на синтетике (валидация 2024-2, 2025-1; базовая частота риска ~17%):

```
PR-AUC (average prec.): ~0.59   (lift к базе ~x3.4)
ROC-AUC               : ~0.87
При пороге топ-10%: recall ~40%, precision ~69%
recall@5% ~24% / recall@10% ~40% / recall@20% ~61%
Калибровка: предсказанная и фактическая частота совпадают по бинам.
```

Числа зависят от seed синтетики; на реальных данных будут другими.

## Этика и безопасность

- Персональные данные **строго внутри сети**, модели локальные. Наружу — ничего.
- Список риска — **не «чёрная метка»**, а инструмент поддержки. В UI и в ответах
  API есть явная пометка `disclaimer`: это вероятностный сигнал, а не приговор.
- Доступ к списку риска **логируется** (`access_log`: кто, что, когда смотрел).
- Защищённые операции (скоринг/обучение/уведомления) требуют сервисный токен.
- Доступ к дашборду — по роли (эдвайзер/декан); запросы идут через серверный
  прокси, минуя прямой доступ браузера к бэкенду.

## Структура проекта

```
backend/
  config.py            конфигурация из переменных окружения
  data_queries.yaml    SQL под Платонус (плейсхолдеры; впишите свои таблицы)
  platonus.py          слой доступа к данным (Платонус / синтетика)
  synthetic_data.py    генератор синтетических данных с метками отчисления
  features.py          признаки с временным срезом без утечки
  model.py             LightGBM: обучение, калибровка, вклад факторов, артефакт
  train.py             переобучение модели + метрики
  evaluate.py          отчёт по качеству (PR-AUC, recall@k, калибровка)
  scoring.py           батч-скоринг → прогнозы в БД
  storage.py           хранилище (Postgres / SQLite)
  llm.py               текст уведомлений (Qwen3) + заглушка
  app.py               FastAPI: /health /risk /student /summary /score/run /train /notify/run
  run_pipeline.py      end-to-end прогон на синтетике
  migrations/001_init.sql   схема таблиц
frontend/              Next.js 15 дашборд (App Router, серверный прокси)
n8n/risk_group_weekly.json  еженедельный воркфлоу
docker-compose.yml     Postgres для прод-подобного запуска
```
